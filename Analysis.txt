The latency test results are overall expected. The US-Central instance saw much lower average times 
because the Firestore database is physically located in the same region (us-central1). Meanwhile, the 
Europe instance experienced 10 times as much latency because server operations have to cross the Atlantic 
Ocean (and back) to communicate with the us-central database. So, even though the two instances used the 
same code, a lack of proximity to the source of data can be an obstacle in global applications.

The consistency experiment results initially surprised me; eventual consistency was expected due to the
architecture, but there weren't any misses even after 100 iterations. To catch a consistency miss, the EU 
read request must arrive at the database before the US write request. However,the Python request library is 
synchronous; so my script waits for the write to be successful before attempting the read, which is enough 
time for Google to internally synchronize the data. The script was too slow to observe the stale state.
